I. Card
1st: CardLogic.js 
CardLogic.js defines the core blueprint for a ygo card; It cointans two classes Card & GameCard;
CardLogic.js splits the card into two distinct concepts but both important for how the game is played:
-> first concept is the static representation of a card (class Card); in simpler terms is the physical traits
of the card (the id, name, type, frameType, description, race, attack, defense, level, attribute),
basically everything that is present on the card (note to myself in the future, add the image of a card for better representation)
-> second concept is the dynamic representation of a card (class GameCard);
It is the card that is being used/played in a game. 
It wraps around the static concept, and in theory should track every change of that card during gameplay;
So when you create a new GameCard(card, owner), it tracks:
    --> the static physical representation of the card, and who owns it;
    --> the location, where it is on the board
(in hand of the owner, in deck waiting to be drawn, on the monster or spell-trap zone, on the graveyard, or in the banished zone, *!fusion zone);
*zoneIndex will be used since the monster + spell-trap zone have only 5 fields;
    **monster cards
    --> the faceUp or faceDown
    --> the position: attack or defense;
    --> for a monster, we should know if it was normal summoned already that turn, if they attacked that turn and if they used their effect or not
    **spell-trap cards
    --> if the card was set
    --> if the cards was activated
    *** this one is a special case for monster that might be changed and deleted; so basically during the gameplay they might get their stats changed, or can't attack or even can't be destroyed, 
    this is still as a reminder in case it gets complicated;
    might need to add even the change attribute due to card DNA Changer
Now you may ask why is written like this? By separating the static from the dynamic of a card, the game doesn't have to reload card text or image URL every time a card moves.
If a monster get's let's say a boost of 500 attack from a equipped spell card, the game updates only GameCard.modifiers.atk = 500, while the original Card.atk remaines safely untouched by this modify.

The JSON format is made of: mainDeck that contains an array of cards, as well the extraDeck and sideDeck;

=================================================================================================

deckBuilder.js with it's function buildDeckForPlayer(json_deck, player),
this function maps the json into: Arrays of objects: mainDeck, extraDeck and sideDeck,
the objects are just new Card(card) that is wrapped inside GameCard(card, player);
                1. RAW DATA (from JSON)
                { "name": "Ryu-Kishin Powered", "atk": 1600, ... }
                        │
                        ▼
                2. NEW CARD (The Blueprint)
                Card { name: "Ryu-Kishin Powered", atk: 1600 }
                        │
                        ▼
                3. NEW GAMECARD (The Game Piece)
                GameCard {
                    card: Card { name: "Ryu-Kishin Powered", atk: 1600 },
                    owner: Player { name: "Kaiba", ... },
                    location: "deck",
                    faceUp: false,
                    ...
                }
    so this function just returns Arrays of Objects GameCard, that will be used by the next class, class Player;

IV ZoneManager.js
are the hands that move the cards during gameplay;
in tcgs the gameplay is also the flow/movement of the cards,things like:
- drawing a card = move(card,from deck, to hand);
- summoning/setting a monster =if card.card. move(card, from hand, to hand) *faceUp/faceDown  
- setting a trap
- activating a spell card
- battling a monster 

that means ZoneManager shoud: 
- move cards between zones
- remove cards safely
- auto-place into monster/spell zones
- enforce “5-slot limit”
- maintain location + zoneIndex

let's recap:
                  Player - is like a storage a player has when they come to a game
                  ├─ LP
                  ├─ zones (deck + extraDeck initialized with cards)
                  ├─ draw()
                  └─ shuffle()

                  ZoneManager - is like player's hands, that can move cards during a game
                  ├─ removeCard()
                  ├─ addCard()
                  └─ moveCard()

                  MainGame - is like the referee which decides if u can: summon,activate,draw,attack etc during a game
                  ├─ turns
                  ├─ phases
                  ├─ summon rules
                  ├─ battle rules
                  ├─ effect rules
                  └─ uses ZoneManager
